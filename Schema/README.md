# Schema Reference

This project is a **terminal-first Python budgeting app**. It does **not** currently use a relational database or ORM migration system.

The authoritative runtime contracts live in `schemas.py`. This folder documents the current data model and the file formats the app already uses, so future API, storage, or database work has a single place to start.

## Evaluation

### What is already solid

- `budget.py` keeps core allocation and comparison logic separate from the CLI.
- `budget_cli.py` handles interaction and display without owning the business rules.
- `schemas.py` centralizes typed contracts for the app's major data shapes.
- `storage.py` uses atomic writes, which is a good safety choice for local file persistence.

### Gaps this folder helps address

- There was no dedicated schema directory for developers to inspect quickly.
- The project has runtime type contracts, but not a lightweight schema reference for contributors.
- Some docstrings refer to `ledgerlogic.*` package paths while the current repo structure is still mostly root-level modules, so a plain-English schema reference helps reduce confusion.

## Core runtime objects

### `BudgetCategoryProfile`

Represents one budget category.

```python
{
  "tier": "Needs" | "Wants" | "Savings",
  "weight": int | float,
  "priority": int,
  "actual_spend": float,
  "budgeted_amount": float,
}
```

### `BudgetAllocation`

Return shape for the allocation strategies.

```python
{
  "strategy": str,
  "allocations": dict[str, float],
  "categories": dict[str, BudgetCategoryProfile],
  "allocated_total": float,
  "remaining": float,
  "warnings": list[str],
}
```

### `BudgetComparisonRow`

One budget-vs-actual row.

```python
{
  "category": str,
  "budgeted": float,
  "actual": float,
  "difference": float,
  "percentage_of_budget": float | None,
  "status": "OVER" | "UNDER" | "EVEN",
  "tier": "Needs" | "Wants" | "Savings" | "Unknown",
  "priority": int,
}
```

### `BudgetComparisonResult`

Summary output from comparing actuals to a budget.

```python
{
  "rows": list[BudgetComparisonRow],
  "overages": set[str],
  "under_budget": set[str],
  "total_overage": float,
  "total_surplus": float,
  "total_actual": float,
  "total_budgeted": float,
}
```

### `CategorizedRecord`

Normalized transaction row used for analysis or import/export.

```python
{
  "date": str,
  "merchant": str,
  "amount": float,
  "category": str,
  "subcategory": str?,
  "confidence": float?,
  "match_type": str?,
}
```

## Persisted file formats

The app is file-backed today, so these are the practical storage schemas.

### `ledgerlogic_data/categorized_transactions.csv`

Columns:

1. `date`
2. `merchant`
3. `amount`
4. `category`
5. `subcategory`
6. `confidence`
7. `match_type`

Example row:

```csv
date,merchant,amount,category,subcategory,confidence,match_type
2026-04-07,Trader Joe's,54.23,Food,Groceries,0.98,exact
```

### `ledgerlogic_data/budget_profile.json`

A storage path exists for this file in `storage.py`, but the current CLI does not yet fully standardize its persisted payload. Recommended canonical shape for future use:

```json
{
  "income": 5000.0,
  "categories": {
    "Rent": {
      "tier": "Needs",
      "weight": 5,
      "priority": 10,
      "actual_spend": 0.0,
      "budgeted_amount": 2500.0
    }
  },
  "actual_spending": {
    "Rent": 2475.0
  },
  "last_strategy": "50/30/20"
}
```

### `ledgerlogic_data/investment_scenarios.json`

A storage path also exists for investment scenarios. Recommended canonical shape for future use:

```json
{
  "scenarios": [
    {
      "name": "Baseline",
      "initial_principal": 1000.0,
      "annual_rate": 7.0,
      "years": 10,
      "compounding": "annual",
      "contribution_amount": 200.0,
      "contribution_frequency": "monthly",
      "contribution_timing": "end",
      "inflation_rate": 2.5
    }
  ]
}
```

### `ledgerlogic_data/ledgerlogic_report.txt`

Plain-text report output generated by the storage layer.

## Suggested maintenance rule

1. Update `schemas.py` first.
2. Update this document second.
3. Add or update tests for any serialized file format changes.
